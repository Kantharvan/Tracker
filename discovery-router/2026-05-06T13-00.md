## Key decisions
- `ProvisioningService.approveAndAssignRole()` remains as a no-op method (logs "Role assignment handled by AMS policies") rather than being deleted entirely â€” preserves the method signature for future refactoring and makes the architectural shift explicit
- IAS Group Rules configuration (stored in IAS Admin Console, not code) is now documented as the responsibility of the IAS/GSSP/MXP teams, not Discovery Router

## Root cause / findings
- With AMS, roles are no longer assigned per-user by Discovery Router via SCIM; instead IAS Group Rules embed a `userType` claim in the JWT at token issuance time, and downstream CAP services enforce access via DCL policies that read this claim
- The repo previously documented that Group Rules must set `userType` but did not explain *how* or *where* â€” this was an open question (docs/business/ams-usage-decision.md, Q4)
